New Rule SOP
新增規則 — 6 步驟標準流程
符合 PRD v3 AC-18：「新增規則不需修改核心程式碼」
1
設計規則邏輯
定義判斷條件、輸出字段、風險分級。於 Rule_XXXX_Design_vN.md 記錄。
2
建立 Denodo View
命名規範：vw_ce_rule_NNN_xxx。包含 SQL 邏輯與資料定義。
3
Registry 登記
RuleRegistry.json：RULE_ID、View、Phase、IS_ACTIVE、SEVERITY。
4
實作檢核器
rule_engine.py：讀取 Registry，生成標準 OUTPUT 格式。無需改核心邏輯。
5
單元測試
test_rule_NNN.py：10+ 案例涵蓋 PASS/WARN/FAIL 路徑。
6
上線與監控
IS_ACTIVE=TRUE，日報儀表板監控覆蓋率與異常。定期審查。
核心特點：Registry 登記制度
RuleRegistry.json 是唯一的「新規則入口」。加新規則只需：(1) 建 Denodo View，(2) 在 JSON 裡新增一筆物件。rule_engine.py 會自動掃描 Registry 並執行對應的 View。不動核心程式碼，確保系統穩定性與可維護性。
Case Example 1 · v4.3
Rule-001：停產風險可信度（EOL Risk Credibility）
CRITICAL 等級 · Phase I · 整合 CO_SURVEY_STATUS + Target EOL Date 動態覆寫
❶ 優先判斷（共用，無視日期）
C1: CO_SURVEY_STATUS = 'No EOL Plan' → GREEN EOL_NO_PLAN (HIGH)
C2: Est EOL NULL/空白 → RED EOL_NULL (NONE)
F3: Est EOL 格式無法解析 → RED EOL_FORMAT_ERR (NONE)
❷ 有 Target EOL Date（Python 動態覆寫）
C3/C4: Est EOL ≥ Target EOL → Vendor Update/Blank GREEN EOL_MEET_TARGET; No Response YELLOW EOL_MEET_TARGET_UNCONFIRMED
C5: Target-2yr ≤ Est EOL < Target → YELLOW EOL_NEAR_TARGET
C6/C7: Est EOL < Target-2yr → Vendor Update/Blank RED EOL_SHORT_OF_TARGET; No Response YELLOW EOL_SHORT_UNCONFIRMED
❸ 無 Target EOL（Fallback 基礎檢核）
F4: Est EOL < TODAY → YELLOW EOL_IN_PAST
F5: Vendor Update → GREEN EOL_VALID_CONFIRMED
F6: No Response → YELLOW EOL_VALID_UNCONFIRMED
CO_SURVEY_STATUS → CONFIDENCE_LEVEL 對應
CO_SURVEY_STATUSCONFIDENCERule-C001
No EOL PlanHIGH長期安全，不fallback
Vendor UpdateHIGHStep1採用
Blank/NULLMEDIUMStep1採用，標記MEDIUM
Vendor No ResponseLOWfallback to Step2+

Rule-001 的 CONFIDENCE_LEVEL 是 Rule-C001 Cascade Step 1 的觸發條件——Vendor No Response 會讓 Rule-C001 跳過 PLM 資料，直接嘗試 PLP/中分類/SE API 備援。
Combination Rule · v4.0 NEW
Rule-C001：Component Lifecycle 最佳 Est EOL 組合推估
四階 Cascade 優先序 · 輸出 BEST_EST_EOL + CASCADE_STEP + CONFIDENCE_LEVEL · 待 Rule-007/008 就緒啟用
Step 1：PLM Est EOL（Rule-001）
條件: CO_ESTIMATED_EOL_DATUM NOT NULL AND CO_SURVEY_STATUS ≠ 'Vendor No Response' | 採用: BEST_EST_EOL = CO_ESTIMATED_EOL_DATUM | CONFIDENCE: Vendor Update→HIGH | 其他→MEDIUM | CASCADE_STEP: STEP1_PLM
若不滿足 ↓
Step 2：廠商 PLP 推估（Rule-003）
條件: PLP_EST_EOL NOT NULL AND Lifecycle_Stage ∉ {EOL, Decline} | 採用: BEST_EST_EOL = PLP_EST_EOL | CONFIDENCE: MEDIUM | CASCADE_STEP: STEP2_PLP
↓
Step 3：中分類平均壽命（Rule-007）★NEW
條件: CATE_M_EST_EOL NOT NULL | 採用: BEST_EST_EOL = CATE_M_EST_EOL | CONFIDENCE: LOW | CASCADE_STEP: STEP3_CATE_M
↓
Step 4：SiliconExpert API（Rule-008）★NEW
條件: SE_EST_EOL NOT NULL AND SE_LIFECYCLE_STATUS ≠ 'EOL' | 採用: BEST_EST_EOL = SE_EST_EOL | CONFIDENCE: LOW | CASCADE_STEP: STEP4_SE_API
↓
Fallback
BEST_EST_EOL = NULL | CONFIDENCE: NONE | CASCADE_STEP: NONE
CONFIDENCE_LEVEL 定義
等級定義來源
HIGHPLM已廠商確認Step1
MEDIUMPLM未確認 或 PLPStep1/2
LOW中分類均值 或 SEStep3/4
NONE四步均無資料Fallback
Rule-C001 的 BEST_EST_EOL 取代 CO_ESTIMATED_EOL_DATUM 用於 BOM Report 的 DIFF(Yrs) 計算。IS_ACTIVE=FALSE，待 Rule-007 + Rule-008 就緒啟用。
RISK_GRADE 總結
BEST_EST_EOL=NULL → 🔴 RED (CASC_NO_DATA)
Vendor Update confirmed → 🟢 GREEN (CASC_OK)
Others → 🟡 YELLOW (CASC_OK, low confidence)

Case Example 2
Rule-002：零件生命週期狀態
CRITICAL 等級 · Phase I · 依 LifeCycle_Phase 值域判斷風險（2026-02 已確認）
LifeCycle_Phase 風險對應
🔴 RED — Phase Out
廠商已宣佈停產計畫；Part Number Release 或 Phase Out 最高警戒。
🔴 RED — End of Life
已進入 EOL 階段；停產在即或已停止供應。
🟡 YELLOW — NRND
No Recommended for New Design；不建議用於新設計，但可持續供應。
🟡 YELLOW — Declining
供應逐漸萎縮；用量需監控。
🟢 GREEN — Product
成品在 M/P 階段（Production/Main）；供應穩定，無風險。
🟢 GREEN — Part Approve
141 IC/Part 已核准，正常供應中。
